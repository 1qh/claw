## Architecture Change: No Elysia, Pure Next.js (2026-03-24)

Eliminated the Elysia control plane server entirely. Everything runs as Next.js API routes in a single process:

| Route                         | Role                                                                |
| ----------------------------- | ------------------------------------------------------------------- |
| `/api/auth/[...all]`          | Auth via `toNextJsHandler` from `better-auth/next-js`               |
| `/api/chat`                   | Chat via HTTP POST to gateway `/v1/chat/completions` (Bearer token) |
| `/api/events`                 | Agent lifecycle logs via SSE (from gateway WS operator connection)  |
| `/api/sessions`               | List sessions from `chat_messages` table (grouped by session key)   |
| `/api/sessions/[id]/messages` | Load messages for a session from `chat_messages` table              |
| `/api/files`                  | File tree from TigerFS `_workspace` + `_state` tables via Drizzle   |
| `/api/files/[...path]`        | File content from TigerFS tables via Drizzle                        |

**Why this is better:**

- Single process: just `bun dev` starts everything
- Next.js instances are stateless ‚Äî any replica handles any request (better for scaling)
- No Eden Treaty / WebSocket proxy complexity
- Auth cookies work naturally (same origin, same process)

**What was removed:**

- Elysia server (the `packages/control-plane/` package is kept as a gateway utility library exporting `connect.ts`)
- Eden Treaty client (frontend type safety via server export)
- WebSocket proxy (frontend ‚Üî control plane ‚Üî gateway)
- Separate `bun run dev` for control plane

## Chat Transport: HTTP `/v1/chat/completions` over WebSocket `chat.send` (updated 2026-03-26)

**Decision:** Use HTTP `/v1/chat/completions` for chat, NOT WebSocket `chat.send`.

### Critical discovery: both run the full agent

Reading the OpenClaw source ([`src/gateway/openai-http.ts`](https://github.com/openclaw/openclaw/blob/main/src/gateway/openai-http.ts)), the HTTP endpoint calls `agentCommandFromIngress()` ‚Äî **the exact same agent runtime** as WS `chat.send`. Both have tools, workspace access, memory, and the full agent loop.

The difference is how they handle agent responses that produce no text:

- **WS `chat.send`**: relays `chat` events verbatim. If the agent calls tools but produces no text, the `final` event has no `message` field ‚Üí client gets empty response.
- **HTTP `/v1/chat/completions`**: after the agent completes, if no assistant text was emitted (`!sawAssistantDelta`), `resolveAgentResponseText()` extracts text from result payloads and falls back to `‚ÄùNo response from OpenClaw.‚Äù` ([line 584-598](https://github.com/openclaw/openclaw/blob/main/src/gateway/openai-http.ts#L584)). **The HTTP handler guarantees a non-empty response.**

This built-in fallback is why HTTP is 100% reliable while WS drops ~15-25% of responses.

### Comparison

| Aspect                   | HTTP `/v1/chat/completions`                                | WebSocket `chat.send`                                                           |
| ------------------------ | ---------------------------------------------------------- | ------------------------------------------------------------------------------- |
| **Agent runtime**        | Full agent (tools, workspace, memory) ‚Äî same as WS         | Full agent (tools, workspace, memory)                                           |
| **Reliability**          | 100% ‚Äî built-in empty response fallback in handler         | ~75-85% ‚Äî no fallback, empty agent turns = empty response                       |
| **Session state**        | Stateless ‚Äî client sends full history each request         | Same ‚Äî gateway doesn‚Äôt inject history (see [limitations.md](../limitations.md)) |
| **Streaming**            | SSE with 150ms throttle (same as WS)                       | WS events with 150ms throttle                                                   |
| **Agent logs**           | Not available via HTTP ‚Äî separate SSE `/api/events` needed | Same WS connection carries chat + agent events                                  |
| **AI SDK compatibility** | Standard OpenAI-compatible ‚Äî works with `@ai-sdk/openai`   | Requires custom transport                                                       |
| **Auth**                 | Bearer token (gateway password)                            | Operator + password in WS handshake                                             |

### Why HTTP wins

1. **Reliability**: The gateway‚Äôs HTTP handler has a fallback for empty agent responses. The WS protocol doesn‚Äôt. With small models (qwen3.5:4b/9b), the agent sometimes calls tools (reading workspace files like SOUL.md, IDENTITY.md) and ends the turn without producing text. This happens ~15-25% of the time. HTTP catches it; WS doesn‚Äôt.

2. **AI SDK integration**: HTTP `/v1/chat/completions` is OpenAI-compatible, so we use `@ai-sdk/openai` pointed at the gateway. This gives us `streamText` + `toDataStreamResponse()` on the server and `useChat` on the client ‚Äî the standard AI SDK pattern with zero custom code.

3. **Simplicity**: The `/api/chat` route is ~30 lines. The WS version was ~100 lines with event filtering, runId tracking, dual text capture, timeouts, and still wasn‚Äôt reliable.

### Previous WS rationale (now invalidated)

- ‚ÄúServer-side context‚Äù ‚Äî the gateway does NOT inject session history into `chat.send`, so both require client-sent history
- ‚ÄúSingle connection for chat + events‚Äù ‚Äî events still need separate SSE anyway
- ‚ÄúAgent tools enrich responses‚Äù ‚Äî both endpoints run the full agent; the tools are identical

### AI SDK integration (2026-03-26)

```
Browser ‚Üí useChat (@ai-sdk/react + TextStreamChatTransport) ‚Üí POST /api/chat ‚Üí streamText(gateway.chat()) ‚Üí text stream
Browser ‚Üí GET /api/events (SSE) ‚Üí Next.js route ‚Üí WS operator connection ‚Üí forward agent events as SSE
```

Server route uses `createOpenAI({ baseURL: gateway, apiKey: password })` + `streamText({ model: gateway.chat(modelId) })` + `toTextStreamResponse()`. **Important:** `@ai-sdk/openai` v3+ defaults to the Responses API (`/v1/responses`). Must use `gateway.chat()` (not `gateway()`) to force chat completions endpoint. Client uses `useChat` with `TextStreamChatTransport({ api: ‚Äò/api/chat‚Äô })` and `body: { sessionKey }`. `append()` for sending, `setMessages()` for session switching, `onFinish` for post-response actions.

**Note:** `packages/control-plane/connect.ts` is still used by `/api/events` for the WS operator connection. It‚Äôs no longer used by `/api/chat`.

### What doesn‚Äôt change

- Auth: still better-auth via `/api/auth/[...all]`
- Session list: reads from `chat_messages` table (grouped by session key, ordered by most recent)
- Session messages: reads from `chat_messages` table (both user and assistant messages)
- JSONL transcripts store only assistant responses ‚Äî user messages don‚Äôt exist in OpenClaw‚Äôs storage

## Device Identity Removed (2026-03-25)

Ed25519 device identity was unnecessary for our use case. OpenClaw‚Äôs [`roleCanSkipDeviceIdentity`](https://github.com/openclaw/openclaw/blob/main/src/gateway/role-policy.ts) returns `true` when `role === "operator" && sharedAuthOk`. Since we connect with operator role + gateway password, device identity is not needed.

**Removed:**

- Device keypair generation and storage
- Ed25519 signing in WS handshake
- `DEVICE_IDENTITY_PATH` env var
- `device` field in WS connect params
- The entire pairing/approval problem

**Cleaned up:** `device-identity.ts` file deleted.
